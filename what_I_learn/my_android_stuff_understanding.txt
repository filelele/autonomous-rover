On Linux, app runs: 
    Linux Kernel -> Process -> main()

On Android, app runs: 
    Linux Kernel -> Android Framework -> Activity Manager Service (AMS) -> main Java/Kotlin UI thread 
                                                                        -> Activity object created

Main UI thread constantly loop to get Lifecycle Events (Commands) from AMS like:

    + APP_CMD_START //User start app
    + APP_CMD_INIT_WINDOW
    + APP_CMD_TERM_WINDOW
    + APP_CMD_STOP 
    + APP_CMD_DESTROY //User quit app
   then call corresponding object function Activity.on(X)Command() to handle.

Android apps that use C++ for high peformance, create NativeActivity object instead.

NativeActivity object when initialize load the compiled C++ file (in .so format), then create a separate C++ thread.

C++ thread constantly listen to Events(include Lifecycle Commands) indirectly through Pipe, by ALooper_pollOnce():
    
    '''
    while (true) {
        int ident; // >= 0 there is new Event, < 0 no new Event and skip
        int events; //whether the read success or error
        struct android_poll_source* source; //store the new Event object memory region address (any type of event: Lifecycle, touch, etc...)
        
        // 0 milliseconds time out = no blocking, no command then skip no wait!
        while ((ident = ALooper_pollOnce(0, nullptr, &events, (void**)&source)) >= 0) {
            
            // Process Events (screen flips, touch, pausing, anything, state->onAppCmd(), state->onTouch(), etc...)
            if (source != nullptr) {
                source->process(state, source); //eg. run state->onAppCmd internally, if the event is a Lifecycle event.
            }

            // state->onAppCmd(state, APP_CMD_DESTROY) only clean resources, cannot stop this c++ thread.
            // So use this for flow control, to stop this c++ thread when meet Lifecycle event APP_CMD_DESTROY
            if (state->destroyRequested != 0) {
                LOGI("Exiting C++ loop.");
                return;
            }
        }

        // Thread logic here
    }
        ...
    '''

Android apps have no Terminal attached to it -> stdout/stderr lead to /dev/null -> use __android_log_print write to logd.
* Log for dev, Screen for user, so write log to logd daemon.
